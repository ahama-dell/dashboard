[2026년 06월 25일 대화 요약 - 실시간 연동 강화 및 통신 구조 분석]

1. proxy_server.py 백그라운드 무한루프 및 세션 핑(Ping) 강화
- 문제점: 기존 파이썬 프록시 서버는 세션이 만료되거나 네트워크 오류 발생 시 데이터 수집 스레드가 강제 종료(return)되어, 서버를 재시작하기 전까지 데이터 수집이 완전히 멈추는 치명적인 버그가 있었음.
- 해결: 
  - 어떤 에러가 발생해도 스레드가 죽지 않고 1분 대기 후 지속적으로 재로그인을 시도하는 무한 루프(while True) 구조로 완벽히 개조함.
  - 세션 유지를 위한 핑(Ping) 간격을 2.5분에서 1분 단위로 촘촘하게 단축하여 세션 만료를 원천 차단함.
  - 10분 대기 시 통으로 대기하지 않고 1초 단위로 감시하며 대기하도록 수정.

2. tempcare.kr 실시간 통신 프로토콜 분석
- 사용자 문의: tempcare.kr 사이트처럼 원천 데이터를 실시간으로 가져오는 방법 문의.
- 브라우저 개발자 도구(F12) 분석 결과:
  - Socket(WS) 탭에 데이터가 없는 것으로 보아, 공식 사이트도 진정한 실시간(웹소켓)을 사용하지 않음.
  - Fetch/XHR 탭에서 'logs' API를 주기적으로 호출하는 것을 확인. 즉, 우리가 만든 proxy_server.py의 통신 방식이 공식 사이트와 100% 동일한 가장 원천적인 방식이었음.

3. 데이터 수집 간격(INTERVAL_MIN) 단축 (10분 -> 2분)
- 사용자가 캡처한 공식 사이트 설정 화면에서 하드웨어 센서의 데이터 전송 주기가 '2분' 간격임을 확인.
- 이에 맞춰 proxy_server.py의 수집 주기를 10분에서 2분으로 단축(INTERVAL_MIN = 2).
- temp.html(프론트엔드) 역시 2분(120,000ms) 주기로 프록시에 요청하도록 세팅되어 있어, 톱니바퀴처럼 완벽히 동기화됨.

4. 시놀로지(Synology) NAS 작업 스케줄러 재시작 안내
- 사용자가 PC에서 수정한 코드가 적용되지 않고 대시보드 화면이 멈춰있던 원인 분석.
- 파이썬 서버를 시놀로지 NAS에서 백그라운드로 돌리고 있었기 때문에, OneDrive로 파일이 동기화되더라도 NAS 메모리 상의 프로세스는 구버전 코드로 돌고 있었음.
- 해결: NAS 제어판의 작업 스케줄러에서 미리 만들어둔 'Kill_Proxy'를 실행해 구버전을 종료하고, 'Tempcare_Proxy_Server'를 다시 실행하여 최신 코드를 로드하도록 안내함.
